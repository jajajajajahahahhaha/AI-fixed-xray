## 🐛 ۱) رفع باگ غیرفعال شدن هسته Xray

### مشکل قبلی
- در تابع `sync_xray_core` بعد از `fuser -k` هیچ صبری برای رها شدن پورت‌ها وجود نداشت
- فقط **یک بار** تلاش برای راه‌اندازی Xray می‌شد و اگر شکست می‌خورد، هیچ چیز آن را دوباره اجرا نمی‌کرد
- **هیچ سیستم watchdog** نبود که Xray را زنده نگه دارد

### راه‌حل
1. در `sync_xray_core`:
   - قبل از راه‌اندازی مجدد، از `pkill -f 'xray -config'` برای کشتن تمیز استفاده می‌شود
   - **1.5 ثانیه** صبر می‌کند تا پورت‌ها آزاد شوند
   - **تا 3 بار** تلاش برای راه‌اندازی می‌کند
   - هر تلاش با `is_xray_core_running()` تایید می‌شود

2. **تابع جدید `xray_watchdog`** به عنوان یک thread پس‌زمینه:
   - هر **6 ثانیه** وضعیت Xray را چک می‌کند
   - اگر Xray down باشد، **خودکار** آن را restart می‌کند
   - بعد از 3 شکست پشت‌سرهم، **کل config را از نو می‌سازد**
   - در هر بار بازیابی، `iptables` block ها دوباره اعمال می‌شوند

### فایل‌های تغییر یافته
- `analytics_worker.py`:
  - تابع `sync_xray_core` بهبود یافته (خط ~980)
  - تابع جدید `xray_watchdog` اضافه شد (خط ~1040)
  - Thread watchdog در startup اجرا می‌شود (خط ~3624)

## 🛡️ ۲) سیستم قوی بلاک IP (چند لایه)

### مشکل قبلی
- سیستم قبلی فقط با تنظیم `active=False` **کل کاربر** را غیرفعال می‌کرد وقتی از max_ips عبور می‌کرد
- **هیچ لیست بلاک IP** در سطح firewall یا Xray routing نبود
- یک IP بلاک شده می‌توانست با کاربر دیگری وصل شود

### راه‌حل — بلاک چند لایه‌ای

#### 🔒 لایه ۱: `iptables` (سطح Kernel)
- استفاده از `iptables -I INPUT -s <IP> -j DROP`
- استفاده از `iptables -I FORWARD -s <IP> -j DROP`
- **قوی‌ترین لایه** — حتی قبل از رسیدن به Xray بلاک می‌شود
- تابع: `apply_iptables_block(ip)` / `remove_iptables_block(ip)`

#### 🔒 لایه ۲: Xray Routing Rules
- در config Xray، یک outbound `blackhole` با تگ `blocked_out` اضافه می‌شود
- IP‌های بلاک شده به‌عنوان `source` در routing rule قرار می‌گیرند
- **پشتیبان** — اگر iptables کار نکرد، خود Xray جلوی اتصال را می‌گیرد

#### 🔒 لایه ۳: قطع فوری اتصالات فعال
- با `ss -K dst <IP>` و `ss -K src <IP>` تمام اتصالات فعال از/به آن IP قطع می‌شوند
- **بلاک شامل حال هم می‌شود**، نه فقط اتصالات جدید

#### 🔒 لایه ۴: در لاگ sniffer
- در `xray_live_log_sniffer` قبل از هر پردازش، `is_ip_blocked(ip)` چک می‌شود
- IP‌های بلاک شده **کاملاً نادیده گرفته می‌شوند** (بایت‌هایشان حساب نمی‌شود)

#### 🔒 لایه ۵: پایداری بعد از restart
- لیست IP‌های بلاک در فایل `blocked_ips.json` ذخیره می‌شود
- در startup پنل، `reapply_all_iptables_blocks()` اجرا می‌شود
- در هر restart Xray، بلاک‌ها دوباره اعمال می‌شوند

### فایل‌های جدید
- `blocked_ips.json`: لیست IP‌های بلاک شده (خودکار ساخته می‌شود)
- `ip_traffic.json`: تاریخچه ترافیک هر IP

## 🌐 ۳) تب جدید مدیریت IP (🌐)

### قابلیت‌ها
تب جدید در navbar بین 🛡️ و بقیه تب‌ها اضافه شده است. **هیچ تب یا قسمت دیگری تغییر نکرده.**

#### 📊 داشبورد بالای تب
- تعداد کل IP‌های متصل الان
- تعداد کل IP‌های بلاک شده

#### 🔵 تب فرعی "متصل"
برای هر IP فعال نشان می‌دهد:
- **آدرس IP** (با فونت مونو، LTR)
- **کاربری که به آن وصل است** (به‌صورت واضح)
- **حجم مصرفی** آن IP
- **تعداد سایت‌های بازدید شده** + **لیست نام سایت‌ها** (به‌صورت tag)
- **تعداد اتصالات**
- **آخرین فعالیت** (چند ثانیه پیش)
- دکمه قرمز **🚫 مسدود کردن** (با تایید)

#### 🔴 تب فرعی "بلاک شده"
برای هر IP بلاک شده نشان می‌دهد:
- تمام اطلاعات بالا
- **زمان بلاک شدن** به تاریخ فارسی
- **دلیل بلاک**
- دکمه سبز **✅ رفع بلاک** (با تایید)

### به‌روزرسانی خودکار
- وقتی در تب IP Manager هستید، هر **4 ثانیه** لیست تازه می‌شود
- دکمه دستی 🔄 تازه‌سازی هم موجود است

### API‌های جدید
| Endpoint | Method | توضیح |
|---|---|---|
| `/api/ip_connections` | GET | لیست IP‌های فعال + بلاک شده |
| `/` (form) | POST `action=block_ip` | بلاک کردن یک IP |
| `/` (form) | POST `action=unblock_ip` | رفع بلاک یک IP |
| `/` (form) | POST `action=clear_ip_traffic` | پاک کردن تاریخچه IP‌های قدیمی |
